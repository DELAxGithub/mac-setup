オールインワン英語学習アプリ（仮称）要件定義書

1. プロジェクトの目的とビジョン

1.1 背景

従来の英語学習は、固定的な教材による「素材不足」、内容の「退屈さ」、そしてエビングハウスの忘却曲線に基づかない「短期記憶での忘却」という3つの大きな課題を抱えていた。本プロジェクトは、AIと現代のウェブエコシステムを統合することで、日常的に消費するYouTubeやNetflixといった動的なコンテンツを、パーソナライズされた高品質な学習素材へと変換し、これらの課題を根本から解決する。

1.2 ビジョン

本システムは、言語を単なる「知識」ではなく「長期記憶にインストールされたスキル」へと昇華させることを目指す。特に「話す能力（口頭英作文）」と「読む能力（語彙獲得）」にフォーカスし、AI音声認識と間隔反復学習（SRS）を組み合わせることで、脳の仕組みに最適化した「効率的な学習システム」を構築する。

2. システム全体像

本システムは、コンテンツからの素材抽出を担う「Data Extraction Layer」と、モバイル端末での反復学習を担う「Learning & Retention Layer」の2層構造で構成される。

2.1 構成モジュールと役割

* Chrome拡張機能（Data Extraction Layer）
  * YouTube/Netflix等のメディアプラットフォームからの字幕コンテキスト抽出。
  * Google翻訳APIを利用した、瞬間英作文用対訳データの生成。
  * Podcast等の高密度コンテンツに最適化したクリッピングロジック。
* スマートフォンアプリ（Learning & Retention Layer）
  * SM-2アルゴリズム等をベースとしたAnki方式の間隔反復学習。
  * ELSA Speakのモデルに準拠したAI音声認識・発音矯正フィードバック。
  * 画像検索API連携による、視覚的イメージの自動登録。

3. 機能要件：学習素材の抽出と生成（Chrome拡張機能）

3.1 字幕コンテキスト抽出と整形

* [FR-EXT-01] インタラクティブ字幕インターフェース: YouTube/Netflixのメタデータから字幕を取得し、画面右側に検索・スクロール可能なサイドバーとして表示する。
* [FR-EXT-02] コンテキスト抽出・自動ボールド処理: 未知の単語を選択した際、その単語単体ではなく、単語が含まれる「一文全体（フルセンテンス）」を抽出する。その際、UI上での視認性を高め、文脈（コンテキスト）の想起を助けるために、対象語彙を自動的に<b>タグまたはMarkdownの太字（**）で囲む処理を実装する。
* [FR-EXT-03] 高密度コンテンツ・優先ロジック: 効率的な語彙獲得のため、情報密度が高く集中を維持しやすい「Podcastの切り抜き動画」を優先的なソースとして推奨・解析する最適化ロジックを搭載する。

3.2 瞬間英作文用データの生成

* [FR-EXT-04] 直訳ベース翻訳エンジン（Google翻訳優先）: 翻訳エンジンにはGoogle翻訳APIを優先的に採用する。これは、DeepL等の意訳に優れたエンジンではなく、構文の対応関係が明確な「直訳」を取得することで、復習時の日本語→英語への変換（メンタル・リコンストラクション）を容易にするためである。
* [FR-EXT-05] 素材パッケージング: 抽出した英文、ボールド処理済み文脈、直訳日本語、およびオリジナルの音声セグメントを1つの学習カードオブジェクトとしてパッケージングし、バックエンドへ非同期送信する。

4. 機能要件：学習と長期記憶化（スマートフォンアプリ）

4.1 SRS（間隔反復学習）エンジン

* [FR-APP-01] スケジューリング・アルゴリズム: SuperMemo-2 (SM-2) アルゴリズムをベースに、ユーザーの忘却曲線に合わせた動的な復習間隔を設定する。
* [FR-APP-02] 瞬間英作文プロンプト: 学習モードでは「日本語→英語」の順序でカードを提示する。文字を読むだけではなく、日本語を見て即座に口頭で英作文を行う「アクティブリコール」をUI上の基本動作とする。
* [FR-APP-03] Swipe-to-Grade UI: Tinderライクな操作系を採用し、大量のカードを高速処理する。
  * 右スワイプ（Good/Easy）: 正解。次回復習間隔を延長。
  * 左スワイプ（Again）: 不正解。忘却とみなし、学習キューの先頭に再配置。

4.2 視覚・音響補完機能

* [FR-APP-04] 自動画像検索・アタッチ機能: 文字情報（辞書の定義）を介さない直感的な理解を促すため、単語（例：「ledge」）に対して画像検索API（Google Custom Search API等）から最適なイメージを自動取得し、カードに添付する。
* [FR-APP-05] AI音声認識・発音解析: ユーザーの口頭出力を認識し、ネイティブモデルとの比較分析を行う。子音・母音の微細な差異（例：[æ]と[ɑː]の違い）を特定し、視覚的なフィードバックを200ms以下の低遅延で提供する。

5. データ構造と連携要件

5.1 カード構成要素テーブル

システム内で管理される各学習カードのデータ型とソースを以下に定義する。

フィールド名	データ型	ソース	概要
English_Text	String	Extension	抽出された英単語および熟語
Context_Sentence	String	Extension	ターゲットを**でボールド化したフルセンテンス
Literal_Translation	String	Google API	瞬間英作文を助けるための直訳調日本語
Image_Anchor	Blob/URL	Image API	視覚的定着を助ける検索画像
Audio_Segment	Blob/URL	Content/TTS	動画由来の音声またはAI生成音声
Interval_Factor	Float	System	忘却曲線に基づいた学習間隔係数
Next_Review_Date	DateTime	System	次回学習予定日時

5.2 同期要件

Chrome拡張機能で「Save」アクションが実行された際、REST APIを通じてバックエンドDBへ即時格納され、スマートフォンアプリ側の「New Cards」キューにプッシュ通知を介さずシームレスに同期されること。

6. ユーザーエクスペリエンス（UX）設計指針

6.1 習慣化の徹底（Low-Friction Design）

* ストリーク維持の保証: 旅行中や体調不良（ワクチン接種後の発熱時等）でも継続できるよう、1セッションの最小単位を1分程度に設定。オフラインモードを必須とし、どのような環境下でも学習ログを欠かさない設計とする。
* 進捗の可視化: 学習継続日数をダッシュボードに表示し、心理的な報酬系を刺激する。

6.2 アウトプット・ファースト

* 脱・視覚依存: テキストの黙読ではなく、常に「音としての英語」を意識させる。音声認識機能と連動し、音読しなければカードが完了しない「強制音読モード」をデフォルトとする。

7. 制約事項および外部API連携

7.1 外部サービス統合

* YouTube/Netflix Data Access: Language Reactor等の先行事例を参考に、DOM解析による字幕取得ロジックを構築する。
* Google Cloud Translation API: 直訳データ取得用。
* ELSA Speak API (or equivalent): 音素レベルでの発音解析エンジンの統合。

7.2 技術的制約

* CORS制約: コンテンツプラットフォーム側からのクロスドメイン・リソース共有制限に対する適切な回避策（プロキシまたはブラウザ権限の活用）の策定。
* レイテンシ: 音声解析フィードバックは、ユーザーの学習フローを阻害しないよう極めて低遅延で行う必要がある。
* オフライン動作: 基本的なSRSカードの消化および画像・音声データのキャッシュ管理。
